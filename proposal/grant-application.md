# Grant Application Draft

## Project Title

Autonomous Agents for Fraud Detection in Compliance Infrastructure

## One Descriptive Sentence

We propose a defensive-security research project that uses synthetic datasets, simulated evidence pipelines, and narrow autonomous agent roles to detect integrity failures in automated compliance workflows.

## Problem Statement

Compliance programs increasingly rely on automatically generated evidence from CI/CD systems, cloud platforms, ticketing tools, identity systems, and internal workflow software. This shift improves scale, but it also changes the risk profile of audit and assurance work. The problem is no longer limited to whether evidence exists at collection time. It also includes whether downstream evidence remains trustworthy after ingestion, transformation, storage, review, and reuse.

In practice, integrity failures can arise when evidence is tampered with after collection, generated by an unauthorized source, assigned an inaccurate timestamp, injected into a reporting pipeline without a valid provenance chain, or reused outside the context in which it was originally created. These failures can distort compliance reviews, create false confidence for auditors and operators, and make incident reconstruction harder when records are challenged later.

Existing controls often emphasize access management and retention, but they are less explicit about detecting subtle integrity failures across automated evidence pipelines. This project addresses that gap by creating a synthetic research environment for studying fraud and integrity anomalies in compliance infrastructure. The goal is not to claim a finished detector, but to produce a public, defensible framework for dataset design, simulation, agent-based analysis, and benchmark-driven evaluation.

## Project Proposal

This project will build a focused research scaffold for defensive detection of fraud and integrity anomalies in compliance evidence pipelines aligned with environments that may map to frameworks such as SOC 2 and ISO 27001. The repository will include: a proposal package, a Compliance Evidence Integrity Dataset schema, synthetic anomaly examples, safe simulation scenarios, narrow agent role definitions, a threat model, and evaluation criteria.

The technical approach is intentionally limited and reviewer-friendly. First, we define a representative compliance evidence pipeline with explicit trust boundaries across evidence sources, ingestion services, transforms, stores, validation layers, and reviewer consumption. Second, we create synthetic anomaly records that model integrity failures such as evidence tampering, unauthorized generation, timestamp manipulation, pipeline injection, and evidence reuse. Third, we define lightweight research agents with separate responsibilities for evidence analysis, anomaly detection, and verification so that evaluation can focus on task decomposition instead of broad claims about autonomous operation.

The project will then formalize a simulation loop: generate synthetic scenarios, label expected anomalies, run role-specific analysis, and score output quality against benchmark criteria that emphasize precision, provenance reasoning, and safe handling of uncertainty. Public outputs will remain synthetic and defensive in scope. The intent is to create a reproducible public research artifact that makes the grant executable, inspectable, and useful to other defensive researchers.

## Project Timeline

The proposed timeline is six months:

1. Month 1: finalize problem framing, repository scaffold, release guardrails, and dataset schema.
2. Month 2: create synthetic anomaly corpus and scenario catalog.
3. Month 3: define simulation pipeline and narrow agent interfaces.
4. Month 4: implement offline evaluation harnesses and reviewer instructions.
5. Month 5: run initial benchmark passes, document limitations, and refine methodology.
6. Month 6: publish a public research release with benchmark documentation and follow-on recommendations.

## Requested Funding And Resources

Requested support is for a small research buildout rather than production deployment. The budget is intended to cover:

- compute for offline simulation and evaluation
- API credits for controlled model evaluation
- light infrastructure for storage, documentation, and reproducible artifacts
- research and documentation time for dataset curation, threat modeling, and benchmark publication

## Public Benefit And Release Plan

The repository is intended for public release under an open-source license for code and a permissive sharing model for documentation. Dataset materials released publicly will remain synthetic or explicitly anonymized, with no confidential or personal data. Releases will be staged so external reviewers can inspect the schema, methodology, benchmark definitions, and limitations before any code expansion is attempted.
